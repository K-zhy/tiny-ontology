## 附录：Ontology 查询模式 vs Text-to-SQL 对比

### 查询能力对比

| 查询类型 | Ontology (工具调用) | Text-to-SQL | 胜出 |
|----------|-------------------|-------------|------|
| 单表过滤 | ✅ `query_objects` 一次搞定 | ✅ 简单 SELECT | 平 |
| 跨表 JOIN | ✅ 跨 Link 自动编译 | ✅ 直接写 JOIN | 平 |
| 简单聚合 | ✅ `aggregate_objects` | ✅ GROUP BY | 平 |
| 否定查询 (NOT EXISTS) | ❌ 工具无原生支持，LLM 易错 | ✅ 子查询自然表达 | SQL |
| 嵌套子查询 | ❌ 工具间无法传递中间结果 | ✅ 任意嵌套 | SQL |
| 窗口函数 (RANK, PERCENTILE) | ❌ 不支持 | ✅ 原生支持 | SQL |
| 集合运算 (UNION/EXCEPT/INTERSECT) | ❌ 不支持 | ✅ 原生支持 | SQL |
| 递归 CTE | ❌ 不支持 | ✅ 支持 | SQL |
| 多步推理组合 | ⚠️ 依赖 LLM 规划，步骤越多越易错 | ✅ 一条 SQL 表达 | SQL |
| 自由探索式分析 | ❌ 受限于预定义工具 | ✅ 任意 SQL | SQL |

### 工程能力对比

| 维度 | Ontology | Text-to-SQL | 胜出 |
|------|----------|-------------|------|
| **安全性** | ✅ 所有操作经过预定义工具，天然沙箱 | ❌ LLM 生成 SQL 可能注入、越权、删表 | Ontology |
| **写操作** | ✅ Action 有参数校验+事务+审计日志 | ❌ 极难安全支持 INSERT/UPDATE/DELETE | Ontology |
| **权限控制** | ✅ Object/Action 级别配 ACL，粒度细 | ⚠️ 需额外 row-level security，复杂 | Ontology |
| **审计追踪** | ✅ 每步工具调用有语义日志 | ❌ 只有最终 SQL，难解释意图 | Ontology |
| **Schema 变更容忍** | ✅ Ontology 映射层吸收底层变化 | ❌ 表名/列名改了 SQL 就废了 | Ontology |
| **多数据源** | ✅ Object 抽象统一异构源（SQL/API/图） | ❌ 只能查单库或需要 federation | Ontology |
| **可解释性** | ✅ 每步操作有业务语义名称 | ⚠️ SQL 对非技术人员不友好 | Ontology |
| **延迟** | ❌ 3-8 次工具调用，串行 LLM 推理 | ✅ 一次生成一条 SQL | SQL |
| **成本** | ❌ LLM token 消耗大（多轮对话） | ✅ 单次生成，token 少 | SQL |
| **对模型要求** | ❌ 需要强规划+工具使用能力 | ⚠️ 需要 SQL 生成能力 | SQL 略优 |
| **错误恢复** | ✅ 单步失败可重试/换路径 | ❌ SQL 错了只能重新生成 | Ontology |
| **业务逻辑封装** | ✅ Function/Action 封装复杂逻辑 | ❌ 每次都要重新推导 | Ontology |

### 适用场景对比

| 场景 | 推荐方案 | 原因 |
|------|----------|------|
| 企业级 SaaS（权限+审计+写操作） | **Ontology** | 安全性和写操作是刚需 |
| 多源数据整合平台 | **Ontology** | 统一抽象层价值大 |
| 固定业务流程自动化 | **Ontology** | Action 封装 + 工具链稳定 |
| BI 数据分析 / 探索式查询 | **Text-to-SQL** | 查询灵活性决定一切 |
| 临时 ad-hoc 数据提取 | **Text-to-SQL** | 不需要预定义工具 |
| 简单只读数据库问答 | **Text-to-SQL** | 开销低、开发快 |
| 对延迟敏感的实时场景 | **Text-to-SQL** | 一次调用 vs 多轮工具链 |

### 核心取舍

```
         查询灵活性 ←————————————————→ 安全/可控性
              |                              |
        Text-to-SQL                     Ontology
              |                              |
    "能查任何东西"              "只能查允许你查的东西"
    "但可能查出问题"            "但保证每步都安全可审计"
```

> **Ontology 的本质不是"更好的查询引擎"，而是"在 LLM 和数据之间加一层业务语义防火墙"。** 查询能力是被有意限制的——这是 feature 不是 bug。
